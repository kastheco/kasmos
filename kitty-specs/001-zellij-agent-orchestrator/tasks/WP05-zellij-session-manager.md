## Objectives & Success Criteria

**Objective**: Create and manage Zellij sessions programmatically — launch sessions from generated KDL layouts, discover pane IDs by name, handle pane lifecycle operations (open/close/restart), manage focus/zoom, and handle session attach/detach. This is the central runtime module that connects layout generation to actual Zellij execution.

**Success Criteria**:
1. Zellij session creation from KDL layout succeeds and session is accessible
2. Pane IDs are correctly discovered from `zellij action list-panes` output by matching KDL name attributes
3. Panes can be opened, closed, and restarted with correct prompt injection
4. Focus and zoom operations work via `zellij action` CLI
5. Session detection (exists/doesn't exist) works for attach/create decisions
6. Prompt injection failures are detected and reported
7. All Zellij CLI interactions are abstracted behind a trait for testability

## Context & Constraints

- **Crate location**: `crates/kasmos/`
- **Zellij CLI**: Version 0.41+ required (supports `name` attribute, `list-panes`, `WriteCharsToPaneId`)
- **Session naming**: `kasmos-{feature-slug}` (e.g., `kasmos-001-zellij-agent-orchestrator`)
- **Reference**: [plan.md](../plan.md) WP05 section; [spec.md](../spec.md) FR-003, FR-004, FR-013, FR-014
- **Constraint**: All Zellij CLI calls must go through a `ZellijCli` trait to enable mocking in tests
- **Constraint**: Pane discovery must handle race conditions (session creation → panes available delay)
- **Key Zellij commands**:
  - `zellij --layout <path> --session <name>` — create session with layout
  - `zellij list-sessions` — list active sessions
  - `zellij action list-panes` — list panes (requires active session context)
  - `zellij action focus-terminal-pane --pane-id <id>` — focus pane
  - `zellij action toggle-fullscreen` — zoom focused pane
  - `zellij action write-chars-to-pane-id <id> "<text>"` — send text to pane
  - `zellij action close-pane` — close focused pane
  - `zellij attach <name>` — attach to existing session
  - `zellij kill-session <name>` — kill session

   ```rust
   use std::process::Command;
   use std::path::Path;

   /// Trait abstracting Zellij CLI operations for testability.
   #[async_trait::async_trait]
   pub trait ZellijCli: Send + Sync {
       async fn create_session(&self, session_name: &str, layout_path: &Path) -> Result<()>;
       async fn list_sessions(&self) -> Result<Vec<String>>;
       async fn list_panes(&self, session_name: &str) -> Result<Vec<PaneInfo>>;
       async fn focus_pane(&self, session_name: &str, pane_id: u32) -> Result<()>;
       async fn toggle_fullscreen(&self, session_name: &str) -> Result<()>;
       async fn write_to_pane(&self, session_name: &str, pane_id: u32, text: &str) -> Result<()>;
       async fn close_pane(&self, session_name: &str, pane_id: u32) -> Result<()>;
       async fn kill_session(&self, session_name: &str) -> Result<()>;
       async fn attach_session(&self, session_name: &str) -> Result<()>;
       async fn session_exists(&self, session_name: &str) -> Result<bool>;
   }

   #[derive(Debug, Clone)]
   pub struct PaneInfo {
       pub pane_id: u32,
       pub name: String,
       pub is_focused: bool,
       pub exit_code: Option<i32>,
   }

   /// Real implementation that shells out to the zellij binary.
   pub struct RealZellijCli {
       binary: String,  // Path to zellij binary
   }

   impl RealZellijCli {
       pub fn new(binary: &str) -> Self {
           Self { binary: binary.to_string() }
       }

       /// Run a zellij command and capture output.
       async fn run_command(&self, args: &[&str]) -> Result<String> {
           let output = tokio::process::Command::new(&self.binary)
               .args(args)
               .output()
               .await
               .map_err(|e| ZellijError::CreateFailed(e.to_string()))?;

           if !output.status.success() {
               let stderr = String::from_utf8_lossy(&output.stderr);
               return Err(ZellijError::CreateFailed(stderr.to_string()).into());
           }

           Ok(String::from_utf8_lossy(&output.stdout).to_string())
       }
   }

   #[async_trait::async_trait]
   impl ZellijCli for RealZellijCli {
       async fn create_session(&self, session_name: &str, layout_path: &Path) -> Result<()> {
           self.run_command(&[
               "--layout", &layout_path.display().to_string(),
               "--session", session_name,
           ]).await?;
           tracing::info!(session = session_name, "Zellij session created");
           Ok(())
       }

       async fn session_exists(&self, session_name: &str) -> Result<bool> {
           let sessions = self.list_sessions().await?;
           Ok(sessions.iter().any(|s| s == session_name))
       }

       // ... implement other trait methods
   }
   ```

### Subtask T027 – Launch Initial Wave Panes

**Purpose**: After session creation, trigger the initial wave of work packages. This connects the session manager to the wave engine.

**Steps**:

1. Create `crates/kasmos/src/session.rs`:
   ```rust
   pub struct SessionManager {
       cli: Box<dyn ZellijCli>,
       session_name: String,
       pane_map: HashMap<String, u32>,  // WP ID → Pane ID
   }

   impl SessionManager {
       pub fn new(cli: Box<dyn ZellijCli>, feature_slug: &str) -> Self {
           Self {
               cli,
               session_name: format!("kasmos-{}", feature_slug),
               pane_map: HashMap::new(),
           }
       }

       /// Create session and discover initial panes.
       pub async fn start(&mut self, layout_path: &Path) -> Result<()> {
           // 1. Check if session already exists
           if self.cli.session_exists(&self.session_name).await? {
               return Err(ZellijError::SessionExists {
                   name: self.session_name.clone(),
               }.into());
           }

           // 2. Create session with layout
           self.cli.create_session(&self.session_name, layout_path).await?;

           // 3. Wait for panes to initialize (race condition mitigation)
           tokio::time::sleep(std::time::Duration::from_millis(1000)).await;

           // 4. Discover pane IDs
           self.discover_panes().await?;

           Ok(())
       }
   }
   ```

### Subtask T028 – Pane ID Discovery (list-panes + Name Matching)

**Purpose**: Parse `zellij action list-panes` output to build a mapping from WP names (set via KDL `name` attribute) to numeric pane IDs.

**Steps**:

1. Add to `crates/kasmos/src/zellij.rs` (in RealZellijCli):
   ```rust
   async fn list_panes(&self, session_name: &str) -> Result<Vec<PaneInfo>> {
       // Run: zellij action list-panes (within session context)
       // The --session flag or ZELLIJ_SESSION env var may be needed
       let output = self.run_command(&["action", "list-panes"]).await?;
       Self::parse_list_panes_output(&output)
   }

   /// Parse the tab-separated output of `zellij action list-panes`.
   /// Output format varies by version but typically includes:
   /// PaneId, Title/Name, Rows, Cols, ...
   fn parse_list_panes_output(output: &str) -> Result<Vec<PaneInfo>> {
       let mut panes = Vec::new();
       for line in output.lines() {
           let parts: Vec<&str> = line.split('\t').collect();
           if parts.len() >= 2 {
               let pane_id = parts[0].trim().parse::<u32>()
                   .map_err(|_| ZellijError::PaneOperation(
                       format!("Failed to parse pane ID from: {}", parts[0])
                   ))?;
               let name = parts[1].trim().to_string();
               panes.push(PaneInfo {
                   pane_id,
                   name,
                   is_focused: false,  // Parse from output if available
                   exit_code: None,
               });
           }
       }
       Ok(panes)
   }
   ```

2. Add to `SessionManager`:
   ```rust
   impl SessionManager {
       /// Discover pane IDs by matching KDL name attributes.
       /// Retries up to 3 times with 500ms backoff for race condition handling.
       async fn discover_panes(&mut self) -> Result<()> {
           let max_retries = 3;
           let backoff = std::time::Duration::from_millis(500);

           for attempt in 0..max_retries {
               let panes = self.cli.list_panes(&self.session_name).await?;

               self.pane_map.clear();
               for pane in &panes {
                   if pane.name.starts_with("WP") || pane.name == "controller" {
                       self.pane_map.insert(pane.name.clone(), pane.pane_id);
                   }
               }

               if !self.pane_map.is_empty() {
                   tracing::info!(
                       panes = self.pane_map.len(),
                       "Discovered {} panes", self.pane_map.len()
                   );
                   return Ok(());
               }

               tracing::warn!(attempt = attempt + 1, "No panes discovered, retrying...");
               tokio::time::sleep(backoff).await;
           }

           Err(ZellijError::PaneOperation("Failed to discover any panes after 3 attempts".into()).into())
       }

       /// Get pane ID for a WP.
       pub fn get_pane_id(&self, wp_id: &str) -> Option<u32> {
           self.pane_map.get(wp_id).copied()
       }
   }
   ```

### Subtask T031 – Session Attach/Detach Handling

**Purpose**: Handle the case where kasmos is started but a session already exists (reattach) or where the operator detaches and later reattaches.

**Steps**:

1. Add to `SessionManager`:
   ```rust
   impl SessionManager {
       /// Check if our session already exists and handle accordingly.
       pub async fn ensure_session(&mut self, layout_path: &Path) -> Result<SessionState> {
           if self.cli.session_exists(&self.session_name).await? {
               tracing::info!(session = %self.session_name, "Existing session found, reattaching");
               // Rediscover panes (they may have changed)
               self.discover_panes().await?;
               Ok(SessionState::Reattached)
           } else {
               self.start(layout_path).await?;
               Ok(SessionState::Created)
           }
       }

       /// Kill the session (for cleanup or abort).
       pub async fn kill_session(&self) -> Result<()> {
           self.cli.kill_session(&self.session_name).await?;
           tracing::info!(session = %self.session_name, "Session killed");
           Ok(())
       }
   }

   #[derive(Debug)]
   pub enum SessionState {
       Created,
       Reattached,
   }
   ```

### Subtask T032 – Prompt Injection Failure Detection

**Purpose**: Detect when OpenCode fails to start or the prompt injection fails, so the operator can be notified and the WP can be retried.

**Steps**:

1. Add to `SessionManager`:
   ```rust
   impl SessionManager {
       /// Check if a pane's process is still running (i.e., OpenCode started successfully).
       /// This is a heuristic check — if the pane exits quickly after creation,
       /// it likely means the prompt injection or OpenCode launch failed.
       pub async fn check_pane_health(&self, wp_id: &str) -> Result<PaneHealth> {
           let pane_id = self.pane_map.get(wp_id)
               .ok_or_else(|| PaneError::NotFound { wp_id: wp_id.to_string() })?;

           let panes = self.cli.list_panes(&self.session_name).await?;
           let pane = panes.iter().find(|p| p.pane_id == *pane_id);

           match pane {
               Some(p) if p.exit_code.is_some() => {
                   tracing::warn!(
                       wp_id = wp_id,
                       exit_code = ?p.exit_code,
                       "Pane exited — possible prompt injection failure"
                   );
                   Ok(PaneHealth::Exited { exit_code: p.exit_code })
               }
               Some(_) => Ok(PaneHealth::Running),
               None => Ok(PaneHealth::Missing),
           }
       }
   }

   #[derive(Debug)]
   pub enum PaneHealth {
       Running,
       Exited { exit_code: Option<i32> },
       Missing,
   }
   ```

2. Call this check shortly after pane launch (e.g., 2 seconds delay) to catch immediate failures

|------|--------|------------|

| Race condition: panes not ready after session creation | Medium | Retry with backoff (3 attempts, 500ms) in discover_panes |

| Prompt injection failure goes undetected | Medium | Health check 2s after launch, monitor for early pane exits |
