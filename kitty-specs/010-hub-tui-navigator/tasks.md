# Work Packages: Hub TUI Navigator

**Inputs**: Design documents from `/kitty-specs/010-hub-tui-navigator/`
**Prerequisites**: plan.md (required), spec.md (user stories), research.md, data-model.md, quickstart.md

**Tests**: Unit tests for feature scanning, state derivation, action resolution, and CLI parsing. Integration tests for CLI backward compatibility.

**Organization**: Fine-grained subtasks (`Txxx`) roll up into work packages (`WPxx`). Each work package is independently deliverable and testable.

**Prompt Files**: Each work package references a matching prompt file in `/kitty-specs/010-hub-tui-navigator/tasks/` generated by `/spec-kitty.tasks`. Treat this file as the high-level checklist; keep deep implementation detail inside the prompt files.

## Subtask Format: `[Txxx] [P?] Description`
- **[P]** indicates the subtask can proceed in parallel (different files/components).
- Include precise file paths or modules.

## Path Conventions
- **Single crate binary**: `crates/kasmos/src/`
- **Tests**: `crates/kasmos/src/` (inline `#[cfg(test)]` modules)

## Work Package WP01: CLI Restructuring & TUI Plumbing Extraction (Priority: P0)

**Goal**: Make the CLI subcommand optional so bare `kasmos` dispatches to the hub, and extract shared terminal plumbing from `tui/mod.rs` as `pub` functions for reuse by the hub module.
**Independent Test**: `cargo build` succeeds. `kasmos --help` shows updated text. `kasmos list` still works. `setup_terminal`/`restore_terminal`/`install_panic_hook` are `pub` on the `tui` module.
**Prompt**: `/kitty-specs/010-hub-tui-navigator/tasks/WP01-cli-restructuring-tui-plumbing.md`

### Included Subtasks
- [ ] T001 Make `Commands` optional in `Cli` struct (`crates/kasmos/src/main.rs`)
- [ ] T002 Add `hub` module declaration stub (`crates/kasmos/src/hub/mod.rs`, `crates/kasmos/src/lib.rs`)
- [ ] T003 Wire `None` match arm to `hub::run()` placeholder in `main.rs`
- [ ] T004 Extract `setup_terminal`, `restore_terminal`, `install_panic_hook` to `pub` in `crates/kasmos/src/tui/mod.rs`
- [ ] T005 Update `after_help` text in CLI to document hub as default behavior

### Implementation Notes
- `Option<Commands>` is the idiomatic clap pattern for optional subcommands (AD-001).
- The `hub::run()` placeholder returns `Ok(())` immediately -- real implementation comes in WP03.
- Extracting TUI plumbing to `pub` is a visibility change only -- no logic changes.
- Must preserve all existing subcommand behavior (FR-019).

### Parallel Opportunities
- T004 (TUI plumbing extraction) can proceed in parallel with T001-T003 (CLI changes).

### Dependencies
- None (starting package).

### Risks & Mitigations
- Breaking existing CLI behavior: Run `cargo test` and manually verify `kasmos list`, `kasmos --help`.

## Work Package WP02: Feature Scanner (Priority: P0)

**Goal**: Implement `hub::scanner::FeatureScanner` that reads `kitty-specs/` and produces `Vec<FeatureEntry>` with spec status, plan status, task progress, and orchestration status.
**Independent Test**: Unit tests pass for all `FeatureEntry` state combinations. Scanner correctly handles missing directories, empty specs, partial tasks, and stale lock files.
**Prompt**: `/kitty-specs/010-hub-tui-navigator/tasks/WP02-feature-scanner.md`

### Included Subtasks
- [ ] T006 Define `FeatureEntry`, `SpecStatus`, `PlanStatus`, `TaskProgress`, `OrchestrationStatus` types in `crates/kasmos/src/hub/scanner.rs`
- [ ] T007 Implement `kitty-specs/` directory scanning and feature number/slug parsing
- [ ] T008 Implement `spec.md` existence + non-empty check for `SpecStatus`
- [ ] T009 Implement `plan.md` existence check for `PlanStatus` and `tasks/WPxx-*.md` scanning with frontmatter lane parsing for `TaskProgress`
- [ ] T010 Implement `.kasmos/run.lock` PID liveness check and Zellij session listing for `OrchestrationStatus`
- [ ] T011 Write unit tests for scanner with filesystem fixtures (all state combinations)

### Implementation Notes
- Reuse `list_specs.rs` patterns for directory scanning and frontmatter extraction.
- PID liveness check reuses the `is_pid_alive()` pattern from `start.rs` (R-005).
- Zellij session listing uses `ZellijCli::list_sessions()` for `kasmos-<feature>` pattern matching.
- Scanner must be `Send + Sync` for use in `tokio::task::spawn_blocking` (AD-007).

### Parallel Opportunities
- T006 (type definitions) and T007 (directory scanning) can proceed in parallel.
- T008, T009, T010 are independent detection modules.

### Dependencies
- Depends on WP01 (hub module stub must exist).

### Risks & Mitigations
- Stale lock files: Always verify PID liveness before reporting "Running".
- Frontmatter parsing failures: Gracefully degrade to `NoTasks` if parsing fails.

## Work Package WP03: Hub App Core - Event Loop, List View, Navigation (Priority: P0)

**Goal**: Implement the hub's main event loop, feature list rendering, and keyboard navigation. The hub launches, displays features from the scanner, and supports j/k navigation, selection, and quit.
**Independent Test**: Hub launches with `cargo run -p kasmos` (no args), displays feature list, responds to j/k/Enter/Esc/Alt+q. Manual refresh with `r` works.
**Prompt**: `/kitty-specs/010-hub-tui-navigator/tasks/WP03-hub-app-core.md`

### Included Subtasks
- [ ] T012 Define `HubView`, `InputMode` enums and `App` struct in `crates/kasmos/src/hub/app.rs`
- [ ] T013 Implement `hub::run()` async event loop in `crates/kasmos/src/hub/mod.rs` using extracted TUI plumbing
- [ ] T014 Implement feature list rendering (ratatui `List` widget with status indicators) in `hub/app.rs`
- [ ] T015 Implement keyboard navigation: j/k up/down, Enter select, Esc back, Alt+q quit in `crates/kasmos/src/hub/keybindings.rs`
- [ ] T016 Implement periodic refresh timer (5s interval via `tokio::time::interval`) with `spawn_blocking` scanner call (AD-007)
- [ ] T017 Implement manual refresh keybinding (`r`) and Zellij-absent read-only mode detection (FR-017)

### Implementation Notes
- Reuse `tui::EventHandler` for crossterm event polling.
- Reuse `tui::setup_terminal()`, `tui::restore_terminal()`, `tui::install_panic_hook()`.
- `ZELLIJ_SESSION_NAME` env var presence determines read-only vs full mode (AD-002).
- Feature list shows: `[NNN] slug [status-indicator]` per line.
- Status indicators: empty spec, no tasks, X/Y done, complete (checkmark), running.

### Parallel Opportunities
- T014 (rendering) and T015 (keybindings) can proceed in parallel once T012 (App struct) exists.

### Dependencies
- Depends on WP01 (TUI plumbing), WP02 (scanner).

### Risks & Mitigations
- Event loop blocking: Ensure scanner runs on `spawn_blocking`, not the main async task.
- Terminal corruption on panic: Panic hook from WP01 handles this.

## Work Package WP05: Action Resolution & Zellij Wrappers (Priority: P1)

**Goal**: Implement `HubAction` resolution from `FeatureEntry` state and create Zellij action wrappers for inside-session pane/tab operations.
**Independent Test**: Unit tests verify correct action availability for every feature lifecycle state. Zellij wrapper functions compile and have correct command construction.
**Prompt**: `/kitty-specs/010-hub-tui-navigator/tasks/WP05-action-resolution-zellij-wrappers.md`

### Included Subtasks
- [ ] T021 Define `HubAction` enum in `crates/kasmos/src/hub/actions.rs`
- [ ] T022 Implement `resolve_actions(entry: &FeatureEntry) -> Vec<HubAction>` that maps feature state to available actions
- [ ] T023 Implement inside-session Zellij wrappers in `crates/kasmos/src/hub/actions.rs`: `open_pane_right()`, `open_new_tab()`, `go_to_tab()`, `query_tab_names()`
- [ ] T024 [P] Implement `NewFeaturePrompt` input mode in `crates/kasmos/src/hub/app.rs` (inline text input for feature name)
- [ ] T025 Write unit tests for action resolution (all state combinations from data-model.md)

### Implementation Notes
- Inside-session Zellij commands use `zellij action ...` without `--session` flag (R-001).
- Action resolution follows the state machine in data-model.md: CreateSpec when empty spec, Plan/Clarify when spec present but no plan, GenerateTasks when plan present but no tasks, Start when tasks ready, Attach when running.
- `NewFeaturePrompt` captures a feature name string, then dispatches `CreateSpec` with the new name.
- Zellij wrappers use `tokio::process::Command` directly (not the `ZellijCli` trait).

### Parallel Opportunities
- T024 (NewFeaturePrompt) can proceed in parallel with T021-T023 (action resolution + Zellij wrappers).

### Dependencies
- Depends on WP02 (FeatureEntry types), WP03 (App struct for InputMode).

### Risks & Mitigations
- Zellij command failures: Wrap in `Result`, display errors as status messages in the TUI.
- Shell metacharacter injection: Validate all user input before passing to Zellij commands.

- [ ] T026 Implement `CreateSpec` action dispatch: `zellij action new-pane --direction right -- ocx oc -- --prompt "/spec-kitty.specify" --agent controller`
- [ ] T027 Implement `Clarify` action dispatch: `zellij action new-pane --direction right -- ocx oc -- --prompt "/spec-kitty.clarify" --agent controller`

- [ ] T030 Wire action dispatch to hub keybindings (Enter on action in list/detail view)

### Implementation Notes

### Risks & Mitigations

- Pane naming conflicts: Use feature number in pane name to avoid collisions.

## Work Package WP07: Implementation Launch & Start Inversion (Priority: P1)

**Goal**: Implement `StartImplementation` and `Attach` actions from the hub, and invert `kasmos start` to default to TUI mode with `--no-tui` opt-out and `--mode` default change to `continuous`.
**Independent Test**: Starting implementation from hub opens a new Zellij tab. `kasmos start <feature>` defaults to TUI. `kasmos start <feature> --no-tui` uses old behavior. `--tui` flag is silently accepted. `--mode` defaults to `continuous`.
**Prompt**: `/kitty-specs/010-hub-tui-navigator/tasks/WP07-implementation-launch-start-inversion.md`

### Included Subtasks
- [ ] T031 Implement `StartContinuous` action: `zellij action new-tab --name "kasmos-<feature>" -- kasmos start <feature>`
- [ ] T032 Implement `StartWaveGated` action: `zellij action new-tab --name "kasmos-<feature>" -- kasmos start <feature> --mode wave-gated`
- [ ] T033 Implement mode selection UX: Enter=continuous (>6 WP confirmation dialog), Shift+Enter=wave-gated (AD-006)
- [ ] T034 Implement `ConfirmDialog` input mode for >6 WP warning in `crates/kasmos/src/hub/app.rs`
- [ ] T035 Implement `Attach` action: `zellij action go-to-tab-name kasmos-<feature>`
- [ ] T036 Add `--no-tui` flag to `Start` command, change `--mode` default to `continuous`, hide `--tui` flag (AD-004) in `crates/kasmos/src/main.rs`
- [ ] T037 Modify `start::run()` to launch orchestration TUI by default, use `zellij attach` only when `--no-tui` is set (`crates/kasmos/src/start.rs`)

### Implementation Notes
- Mode selection UX (AD-006): Enter defaults to continuous. If >6 WPs, show confirmation dialog before proceeding. Shift+Enter always wave-gated.
- `ConfirmDialog` is a modal overlay with y/n keybindings.
- Start inversion (AD-004): Add `no_tui: bool` flag, change mode default from `wave-gated` to `continuous`.
- In `start.rs`, when `!no_tui`: after creating the Zellij session and starting the engine, call `tui::run()` instead of `zellij attach`.
- The `--tui` flag becomes a hidden no-op for backward compatibility.

### Parallel Opportunities
- T031-T035 (hub actions) can proceed in parallel with T036-T037 (start.rs changes).

### Dependencies
- Depends on WP05 (Zellij wrappers), WP06 (action dispatch pattern).

### Risks & Mitigations
- TUI inversion breaking scripts: `--no-tui` provides escape hatch. `--tui` silently accepted.
- Shift+Enter detection: crossterm may report this differently across terminals; test on target platform.

## Work Package WP08: Orchestration TUI Hub Nav & Edge Cases & Polish (Priority: P2)

**Goal**: Add hub navigation keybinding to the orchestration TUI, handle all edge cases (no kitty-specs, no Zellij, narrow terminal, no WPs), and polish the hub UX.
**Independent Test**: Alt+h from orchestration TUI opens/switches to hub tab. All edge cases display appropriate messages. Hub renders correctly at 80x24 minimum.
**Prompt**: `/kitty-specs/010-hub-tui-navigator/tasks/WP08-orchestration-hub-nav-polish.md`

### Included Subtasks
- [ ] T038 Add `Alt+h` keybinding to orchestration TUI (`crates/kasmos/src/tui/keybindings.rs`) that opens/switches to hub tab (AD-005)
- [ ] T039 Implement hub tab detection: query tab names for "hub" or "kasmos-hub", go-to if found, new-tab if not
- [ ] T040 Handle edge case: `kitty-specs/` does not exist -- display "No kitty-specs/ directory found" message
- [ ] T041 Handle edge case: no Zellij session -- read-only mode with warning banner for unavailable actions
- [ ] T042 Handle edge case: feature has no WP files -- show error and suggest task generation when Start is attempted
- [ ] T043 Handle edge case: terminal too narrow for side-by-side split -- warn or fall back to new tab for agent panes
- [ ] T044 Polish: consistent status bar, help footer with keybinding hints, visual distinction for complete features (dimmed/checkmark)

### Implementation Notes
- Alt+h in orchestration TUI uses inside-session Zellij commands (same as hub actions).
- Tab detection: `zellij action query-tab-names` to check for existing hub tab.
- Read-only mode: check `ZELLIJ_SESSION_NAME` env var; if absent, disable all pane/tab actions and show warning.
- Minimum terminal size: 80x24 (NFR-004). Below this, show a "terminal too small" message.
- Status bar: bottom row showing current view, feature count, last refresh time.
- Help footer: keybinding hints (j/k, Enter, Esc, n, r, Alt+q).

### Parallel Opportunities
- T038-T039 (orchestration TUI changes) can proceed in parallel with T040-T044 (hub edge cases/polish).

### Dependencies
- Depends on WP03 (hub app core), WP06 (agent pane launch), WP07 (implementation launch).

### Risks & Mitigations
- Keybinding conflicts with Zellij: Alt+h is unlikely to conflict; verify against Zellij defaults.
- Terminal size detection: Use crossterm `terminal::size()` and handle gracefully.
