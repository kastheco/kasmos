# WP09: MCP Tools — Message & Event System

## Implementation Command

```bash
spec-kitty implement WP09 --base WP07
```

## Objective

Implement `read_messages` and `wait_for_event` MCP tools — the core communication backbone that allows the manager agent to observe worker output. `read_messages` is a non-blocking snapshot; `wait_for_event` is a blocking poll loop that keeps the manager's agentic loop alive while workers execute. The dashboard pane is updated as a side effect of each poll cycle.

## Context

### Architecture Reference
- **Spec**: FR-009 (message-based communication), FR-010 (structured message format)
- **Contracts**: `contracts/kasmos-serve.json` — read_messages and wait_for_event schemas
- **Data Model**: `data-model.md` — KasmosMessage, MessageEvent, DashboardState, DashboardWorker
- **Plan**: `plan.md` — Manager Agentic Loop, Communication Protocol, Dashboard Display

### Prerequisites (from WP07 + WP03)
- `serve/mod.rs` has `KasmosServer` with shared state including `Arc<AtomicUsize>` for message index
- `serve/messages.rs` has `KasmosMessage::parse()` and `MessageEvent` enum
- `serve/dashboard.rs` has `DashboardState` and ANSI formatting
- `serve/tools/read_messages.rs` and `serve/tools/wait_for_event.rs` exist as stubs

### Key Design Decisions
- `wait_for_event` is the most complex tool — it blocks the MCP response while internally polling
- Dashboard updates happen as side effects during the poll loop (not a separate mechanism)
- Message index tracking prevents re-processing of old messages across calls
- msg-log pane is a plain Zellij pane — content retrieved via `zellij action dump-pane`

### T054: Implement `serve/tools/read_messages.rs`

**Purpose**: Non-blocking tool that dumps the msg-log pane, parses structured messages, and returns filtered results.

**Steps**:
1. Define input/output types:
   ```rust
   #[derive(Deserialize, JsonSchema)]
   pub struct ReadMessagesInput {
       pub since: Option<usize>,       // Messages after this index
       pub filter_wp: Option<String>,   // Filter by WP sender
       pub filter_event: Option<String>, // Filter by event type
   }
   ```

2. Implementation flow:
   a. Call `dump_msg_log_pane()` (from T055) to get raw pane content
   b. Split into lines, iterate
   c. For each line, call `KasmosMessage::parse(line, line_index)`
   d. Apply filters:
      - `since`: skip messages with `index <= since`
      - `filter_wp`: match if `sender.starts_with(filter_wp)` (e.g., "WP-01" matches "WP-01-coder")
      - `filter_event`: match if `event` equals the filter (case-insensitive)
   e. Return `{ messages: [...], total: N }`

3. Message indexing:
   - Assign index based on line position in the pane dump (0-based)
   - This is deterministic — same dump position = same index
   - Update the server's `last_known_index` to the highest parsed index

**Files**: `crates/kasmos/src/serve/tools/read_messages.rs` (~80 lines)

**Validation**:
- [ ] Returns empty array when no messages match
- [ ] `since` filter correctly excludes earlier messages
- [ ] `filter_wp` matches sender prefix
- [ ] Non-matching lines (noise, blank lines) are silently skipped

### T056: Implement `serve/tools/wait_for_event.rs`

**Purpose**: The core blocking MCP tool. Enters an async poll loop that dumps the msg-log pane on each cycle, looks for matching events, updates the dashboard, and returns when a match is found or timeout is reached.

**Steps**:
1. Define input type:
   ```rust
   #[derive(Deserialize, JsonSchema)]
   pub struct WaitForEventInput {
       pub filter_event: Option<Vec<String>>,  // Event types to match
       pub filter_wp: Option<String>,           // Only match this WP
       pub timeout_seconds: Option<u64>,        // Max wait (default from config)
       pub poll_interval_seconds: Option<u64>,  // Poll frequency (default from config)
   }
   ```

2. Implementation — the async poll loop:
   ```rust
   pub async fn wait_for_event(
       input: WaitForEventInput,
       server: &KasmosServer,
   ) -> Result<WaitForEventOutput, anyhow::Error> {
       let timeout = Duration::from_secs(
           input.timeout_seconds.unwrap_or(server.config.communication.wait_timeout_seconds)
       );
       let interval = Duration::from_secs(
           input.poll_interval_seconds.unwrap_or(server.config.communication.poll_interval_seconds)
       );
       let start = Instant::now();
       let last_index = server.last_message_index.load(Ordering::Relaxed);

       loop {
           // 1. Dump msg-log pane
           let content = dump_msg_log_pane(&server.config.session.name).await?;

           // 2. Parse new messages since last_index
           let messages = parse_messages_since(&content, last_index);

           // 3. Update dashboard (side effect)
           update_dashboard(&server, &messages).await;

           // 4. Check for matching event
           for msg in &messages {
               if matches_filter(msg, &input.filter_event, &input.filter_wp) {
                   // Update last_index
                   server.last_message_index.store(msg.index, Ordering::Relaxed);
                   return Ok(WaitForEventOutput {
                       status: "event".to_string(),
                       event: Some(msg.clone()),
                       elapsed_seconds: start.elapsed().as_secs(),
                       last_messages: None,
                   });
               }
           }

           // 5. Update last_index to highest seen
           if let Some(last) = messages.last() {
               server.last_message_index.store(last.index, Ordering::Relaxed);
           }

           // 6. Check timeout
           if start.elapsed() >= timeout {
               let recent: Vec<_> = messages.into_iter().rev().take(5).collect();
               return Ok(WaitForEventOutput {
                   status: "timeout".to_string(),
                   event: None,
                   elapsed_seconds: start.elapsed().as_secs(),
                   last_messages: Some(recent),
               });
           }

           // 7. Sleep and repeat
           tokio::time::sleep(interval).await;
       }
   }
   ```

3. Filter matching:
   ```rust
   fn matches_filter(
       msg: &KasmosMessage,
       filter_event: &Option<Vec<String>>,
       filter_wp: &Option<String>,
   ) -> bool {
       if let Some(events) = filter_event {
           let event_str = serde_json::to_string(&msg.event)
               .unwrap_or_default()
               .trim_matches('"')
               .to_string();
           if !events.iter().any(|e| e.eq_ignore_ascii_case(&event_str)) {
               return false;
           }
       }
       if let Some(wp) = filter_wp {
           if !msg.sender.starts_with(wp.as_str()) {
               return false;
           }
       }
       true
   }
   ```

**Files**: `crates/kasmos/src/serve/tools/wait_for_event.rs` (~120 lines)

**Validation**:
- [ ] Returns immediately when a matching event is already in the pane
- [ ] Blocks and returns when a new matching event appears
- [ ] Returns timeout status after the configured ceiling
- [ ] Filter matching works for event types and WP IDs
- [ ] Poll interval is respected (not spinning)

### T058: Implement message index tracking

**Purpose**: Track the last-seen message index across `read_messages` and `wait_for_event` calls to avoid re-processing.

**Steps**:
1. The message index is stored in `KasmosServer` as `Arc<AtomicUsize>`:
   ```rust
   pub struct KasmosServer {
       // ... other fields
       pub last_message_index: Arc<AtomicUsize>,
   }
   ```
   - This was initialized in WP07 (T043)

2. In `read_messages`:
   - After parsing all messages, update `last_message_index` to the highest parsed index
   - The `since` parameter from the caller overrides this (caller tracks their own cursor)

3. In `wait_for_event`:
   - On each poll cycle, only parse messages with index > `last_message_index`
   - After returning (event or timeout), update the index

4. Helper function:
   ```rust
   fn parse_messages_since(content: &str, since_index: usize) -> Vec<KasmosMessage> {
       content.lines()
           .enumerate()
           .filter_map(|(i, line)| {
               if i <= since_index { return None; }
               KasmosMessage::parse(line, i)
           })
           .collect()
   }
   ```

**Files**: `crates/kasmos/src/serve/tools/read_messages.rs`, `wait_for_event.rs` (add ~20 lines total)

**Validation**:
- [ ] Sequential `read_messages` calls don't return duplicate messages
- [ ] `wait_for_event` only matches events it hasn't seen before
- [ ] Index correctly tracks across interleaved read/wait calls

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Pane dump includes terminal noise (cursor positioning, etc.) | Strip ANSI codes before parsing |
| Dashboard flickering during rapid updates | Use full clear-and-rewrite with ANSI escape |
| `wait_for_event` poll loop CPU usage | tokio::time::sleep between cycles (not busy-wait) |
| Message index drift if pane is cleared | Track by content hash as fallback, or accept reset |
