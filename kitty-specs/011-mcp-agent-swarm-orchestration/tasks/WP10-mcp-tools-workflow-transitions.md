## Objective

Implement `workflow_status` and `transition_wp` MCP tools — the workflow state management layer. `workflow_status` gives the manager a complete picture of all WP states and the overall lifecycle phase. `transition_wp` allows the manager to advance WPs through lifecycle lanes with validation and audit logging. These tools read/write spec-kitty task file YAML frontmatter as the source of truth.

### T062: Implement phase detection

**Purpose**: Determine the current lifecycle phase based on artifact presence and WP lane states.

**Steps**:
1. Phase logic (ordered — first match wins):
   ```rust
   fn detect_phase(
       has_spec: bool,
       has_plan: bool,
       has_tasks: bool,
       lane_summary: &LaneSummary,  // { pending, active, for_review, done, blocked, rework }
   ) -> String {
       if !has_spec { return "no_spec".to_string(); }
       if !has_plan { return "spec_only".to_string(); }
       if !has_tasks { return "planned".to_string(); }

       // Has tasks — determine phase from lane states
       let total = lane_summary.total();
       if lane_summary.done == total { return "complete".to_string(); }
       if lane_summary.for_review > 0 { return "reviewing".to_string(); }
       if lane_summary.active > 0 || lane_summary.rework > 0 { return "implementing".to_string(); }
       return "tasked".to_string();  // All pending
   }
   ```

2. The phase values match the contract enum:
   `no_spec`, `spec_only`, `planned`, `tasked`, `implementing`, `reviewing`, `complete`

- [ ] Feature with spec + plan → "planned"
- [ ] Feature with all WPs pending → "tasked"
- [ ] Feature with some WPs active → "implementing"

### T063: Implement `serve/tools/transition_wp.rs`

**Purpose**: Move a WP to a new lifecycle lane with validation, file locking, and audit logging.

**Steps**:
1. Define input type:
   ```rust
   #[derive(Deserialize, JsonSchema)]
   pub struct TransitionWPInput {
       pub wp_id: String,
       pub target_lane: String,  // "pending", "active", "for_review", "done", "rework", "blocked"
       pub reason: Option<String>,
   }
   ```

2. Implementation flow:
   a. Resolve the task file path: `<specs_dir>/<feature>/tasks/<wp_file>.md`
      - Find the file matching the WP ID (e.g., "WP01" → "WP01-*.md")
   b. Acquire advisory file lock (T064)
   c. Read current frontmatter via `parse_frontmatter()`
   d. Validate transition:
      - Current lane → target lane must be allowed by `state_machine.rs`
      - Call the validation function (check its API: likely `is_valid_transition(from, to) -> bool` or similar)
      - If invalid, return `{ valid: false, error: "Transition from X to Y is not allowed" }`
   e. Update the YAML frontmatter in the file:
      - Read file content
      - Find the `lane:` line in the YAML frontmatter section (between `---` delimiters)
      - Replace `lane: <old>` with `lane: <new>`
      - Write the modified content back
   f. Release file lock
   g. Append audit entry (T065)
   h. Return `{ wp_id, from_lane, to_lane, valid: true }`

3. YAML frontmatter update strategy:
   ```rust
   fn update_lane_in_frontmatter(content: &str, new_lane: &str) -> Result<String, anyhow::Error> {
       // Find YAML block between first two "---" lines
       // Replace the line matching `lane: <anything>` with `lane: <new_lane>`
       // Preserve everything else (body, other frontmatter fields)
       let re = regex::Regex::new(r"(?m)^lane:\s*.*$")?;
       let updated = re.replace(content, format!("lane: {}", new_lane));
       Ok(updated.to_string())
   }
   ```
   - Be careful: only replace within the frontmatter section, not in the body
   - Also update the `history:` section if present (append a transition entry)

**Files**: `crates/kasmos/src/serve/tools/transition_wp.rs` (~100 lines)

### T064: Implement flock advisory locking on task file writes

**Purpose**: Prevent concurrent writes to the same task file from multiple kasmos serve instances.

**Steps**:
1. Create a locking helper:
   ```rust
   use std::fs::File;
   use std::os::unix::io::AsRawFd;
   use nix::fcntl::{flock, FlockArg};

   pub struct FileLock {
       file: File,
   }

   impl FileLock {
       pub fn acquire(path: &Path) -> Result<Self, anyhow::Error> {
           let file = File::open(path)?;
           flock(file.as_raw_fd(), FlockArg::LockExclusive)?;
           Ok(Self { file })
       }
   }

   impl Drop for FileLock {
       fn drop(&mut self) {
           let _ = flock(self.file.as_raw_fd(), FlockArg::Unlock);
       }
   }
   ```

2. Use in `transition_wp`:
   ```rust
   let _lock = FileLock::acquire(&task_file_path)?;
   // Read, validate, modify, write
   // Lock released on drop
   ```

3. This is the same pattern as `serve/audit.rs` uses for JSONL writes. Consider extracting to a shared `utils::file_lock` module if the pattern is duplicated.

**Files**: `crates/kasmos/src/serve/tools/transition_wp.rs` or shared util (~30 lines)

**Validation**:
- [ ] Lock is acquired before file read
- [ ] Lock is released after write (on drop)
