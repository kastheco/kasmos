# WP04: CLI Restructuring

## Objective

Replace the current clap CLI with the new command structure: `kasmos [spec-prefix]`, `kasmos serve`, `kasmos setup`, `kasmos list`, `kasmos status`. Remove or feature-gate old commands (start, cmd, attach, stop, tui). This establishes the new binary interface that all subsequent WPs depend on.

## Context

### Current State

The current `main.rs` CLI uses clap with these subcommands:
- `List` — lists feature specs (KEEP)
- `Start { feature, mode }` — starts orchestration (REPLACE with default positional)
- `Status { feature? }` — shows WP progress (KEEP)
- `Cmd { feature?, command }` — sends FIFO commands (REMOVE/GATE)
- `Attach { feature }` — attaches to Zellij session (REMOVE/GATE)
- `Stop { feature? }` — stops orchestration (REMOVE/GATE)
- `Tui { feature }` — launches TUI dashboard (GATE behind `tui` feature)
- Default (None) — runs `hub::run()` (GATE behind `tui` feature)

Module declarations in `main.rs`: `mod attach; mod cmd; mod feature_arg; mod hub; mod list_specs; mod report; mod start; mod status; mod stop; mod tui_cmd;`

### Target State

New CLI structure per plan.md:
```
kasmos                        # Launch session (feature selector)
kasmos <spec-prefix>          # Launch session bound to feature
kasmos serve                  # MCP server (stdio, spawned by OpenCode)
kasmos setup                  # Validate environment, generate configs
kasmos list                   # List unfinished feature specs
kasmos status [feature]       # Show WP progress for a feature
```

New module structure:
- `launch/mod.rs` — entry point for `kasmos [spec-prefix]`
- `serve/mod.rs` — entry point for `kasmos serve`
- `setup/mod.rs` — entry point for `kasmos setup`

### T022: Rewrite clap CLI struct and Commands enum

**Purpose**: Define the new command hierarchy with clap derive macros.

**Steps**:

1. Replace the `Commands` enum in `main.rs` with:
   ```rust
   #[derive(Parser)]
   #[command(
       name = "kasmos",
       version,
       about = "MCP agent swarm orchestrator for Zellij",
       after_help = "\
   \x1b[1mQuick Start:\x1b[0m
     kasmos                              Launch session (feature selector)
     kasmos <spec-prefix>                Launch session for a feature
     kasmos serve                        Run MCP server (stdio)
     kasmos setup                        Validate environment & install profiles
     kasmos list                         List feature specs
     kasmos status [feature]             Show WP progress"
   )]
   struct Cli {
       #[command(subcommand)]
       command: Option<Commands>,
   
       /// Feature spec ID or prefix (e.g., "011")
       /// Used when no subcommand is given
       spec_prefix: Option<String>,
   }
   
   #[derive(Subcommand)]
   enum Commands {
       /// Run the MCP server (stdio transport, spawned by OpenCode)
       Serve,
       /// Validate environment and install agent profiles
       Setup,
       /// List available feature specs
       List,
       /// Show orchestration status
       Status {
           /// Feature directory (optional, auto-detects)
           feature: Option<String>,
       },
   }
   ```

2. **Important clap note**: Having both `command: Option<Commands>` and a positional `spec_prefix: Option<String>` on the same struct requires careful handling. clap will try the subcommand first, then fall back to positional. Test that `kasmos 011` correctly sets `spec_prefix` and that `kasmos serve` correctly routes to `Commands::Serve`.

   If clap conflicts arise, an alternative pattern is:
   ```rust
   enum Commands {
       Serve, Setup, List, Status { feature: Option<String> },
       /// Hidden: handles the default positional argument
       #[command(hide = true, name = "")]
       Launch { spec_prefix: Option<String> },
   }
   ```
   Choose whichever pattern works cleanly. The key constraint is: `kasmos` and `kasmos 011` both invoke the launch flow, while `kasmos serve/setup/list/status` invoke their respective flows.

3. If TUI feature is enabled, optionally add back the gated commands:
   ```rust
   #[cfg(feature = "tui")]
   /// Launch the hub TUI
   Hub,
   #[cfg(feature = "tui")]
   /// Launch TUI dashboard (internal)
   #[command(hide = true)]
   Tui { feature: String },
   ```

**Files**: `crates/kasmos/src/main.rs`

### T027: Wire main.rs dispatch

**Purpose**: Connect the new CLI variants to their module entry points.

**Steps**:

1. Update the `main()` function dispatch:
   ```rust
   #[tokio::main]
   async fn main() -> Result<()> {
       let cli = Cli::parse();
       let _ = kasmos::init_logging();
   
       // Load config (WP02 provides this)
       let config = kasmos::config::KasmosConfig::load()?;
   
       match cli.command {
           None => {
               // Default: launch session
               launch::run(cli.spec_prefix.as_deref(), &config)
                   .await
                   .context("Launch failed")?;
           }
           Some(Commands::Serve) => {
               serve::run()
                   .await
                   .context("MCP server failed")?;
           }
           Some(Commands::Setup) => {
               setup::run(&config)
                   .await
                   .context("Setup failed")?;
           }
           Some(Commands::List) => {
               kasmos::list_specs::run()
                   .context("Failed to list specs")?;
           }
           Some(Commands::Status { feature }) => {
               kasmos::status::run(feature.as_deref())
                   .context("Status failed")?;
           }
           // TUI commands (feature-gated)
           #[cfg(feature = "tui")]
           Some(Commands::Hub) => {
               hub::run().await.context("Hub TUI failed")?;
           }
           #[cfg(feature = "tui")]
           Some(Commands::Tui { feature }) => {
               tui_cmd::run(&feature).await.context("TUI failed")?;
           }
       }
   
       Ok(())
   }
   ```

2. Adjust as needed based on how T022 structured the clap enum. The key behavior:
   - No subcommand → `launch::run(spec_prefix)`
   - `serve` → `serve::run()`
   - `setup` → `setup::run(config)`
   - `list` → `list_specs::run()` (existing)
   - `status` → `status::run()` (existing)

3. Note: `list_specs` and `status` are in `lib.rs`, accessible via `kasmos::list_specs::run()` or through local `mod` import. Choose the cleaner path.
