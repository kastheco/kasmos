# WP06: Zellij Layout Generation & Session Launch

## Objective

Generate the two-tab Zellij session layout (MCP tab + orchestration tab) as KDL, launch or attach to sessions, and implement the full `kasmos [spec-prefix]` flow. This is the "fire-and-forget" launcher: kasmos generates the layout, spawns Zellij, and exits.

## Context

### Target Layout Structure

```
Tab 1: "MCP" (background)
  └── kasmos serve pane

Tab 2: "orchestration" (active, focused)
  ├── manager pane (60%) — OpenCode --agent manager
  ├── msg-log pane (20%) — empty, receives structured messages
  └── dashboard pane (20%) — empty, updated by kasmos serve
```

### KDL Layout Format

Per data-model.md:
```kdl
layout {
    tab name="MCP" {
        pane command="kasmos" {
            args "serve"
        }
    }
    tab name="orchestration" focus=true {
        pane split_direction="vertical" {
            pane name="manager" size="60%" command="opencode" {
                args "-p" "kasmos" "--" "--agent" "manager" "--prompt" "..."
            }
            pane name="msg-log" size="20%"
            pane name="dashboard" size="20%"
        }
    }
}
```

2. Implement the KDL generation. You can use:

   Recommended: String formatting with proper escaping. KDL is simple enough here:
   ```rust
   pub fn generate_session_layout(
       config: &KasmosConfig,
       feature_slug: &str,
       manager_command: &[String],  // ["opencode", "-p", "kasmos", "--", "--agent", "manager", "--prompt", "..."]
   ) -> Result<String> {
       let mut kdl = String::new();
       kdl.push_str("layout {\n");
       
       // MCP tab (background, runs kasmos serve)
       kdl.push_str("    tab name=\"MCP\" {\n");
       kdl.push_str("        pane command=\"kasmos\" {\n");
       kdl.push_str("            args \"serve\"\n");
       kdl.push_str("        }\n");
       kdl.push_str("    }\n");
       
       // Orchestration tab (active, focused)
       kdl.push_str("    tab name=\"orchestration\" focus=true {\n");
       kdl.push_str("        pane split_direction=\"vertical\" {\n");
       
       // Manager pane (60%)
       // Command is the first element, args are the rest
       let cmd = &manager_command[0];
       let args: Vec<String> = manager_command[1..].iter()
           .map(|a| format!("\"{}\"", a.replace('\"', "\\\"")))
           .collect();
       kdl.push_str(&format!("            pane name=\"manager\" size=\"60%\" command=\"{}\" {{\n", cmd));
       if !args.is_empty() {
           kdl.push_str(&format!("                args {}\n", args.join(" ")));
       }
       kdl.push_str("            }\n");
       
       // Msg-log pane (20%) — empty, receives echo commands
       kdl.push_str("            pane name=\"msg-log\" size=\"20%\"\n");
       
       // Dashboard pane (20%) — empty, kasmos serve writes to it
       kdl.push_str("            pane name=\"dashboard\" size=\"20%\"\n");
       
       kdl.push_str("        }\n");
       kdl.push_str("    }\n");
       kdl.push_str("}\n");
       
       Ok(kdl)
   }
   ```

3. Write the generated KDL to a temp file for Zellij to consume:

   ```rust
   use std::io::Write;
   use tempfile::NamedTempFile;
   
   pub fn write_layout_file(kdl: &str) -> Result<NamedTempFile> {
       let mut file = NamedTempFile::with_suffix(".kdl")?;
       file.write_all(kdl.as_bytes())?;
       file.flush()?;
       Ok(file)
   }
   ```

   ```rust
   /// Zellij context information.
   pub enum ZellijContext {
       /// Running outside any Zellij session.
       Outside,
       /// Running inside a Zellij session.
       Inside {
           session_name: String,
       },
   }
   
   /// Detect the current Zellij context.
   pub fn detect_zellij_context() -> ZellijContext {
       match std::env::var("ZELLIJ_SESSION_NAME") {
           Ok(name) if !name.is_empty() => ZellijContext::Inside {
               session_name: name,
           },
           _ => ZellijContext::Outside,
       }
   }
   ```

### T038: Implement session creation (outside Zellij)

**Purpose**: Launch a new Zellij session with the generated layout when running outside Zellij.

**Steps**:

1. In `launch/session.rs`:
   ```rust
   use std::process::Command;
   
   /// Launch a new Zellij session with the given layout file.
   ///
   /// This replaces the current process (exec) or spawns and exits.
   pub fn create_session(
       session_name: &str,
       layout_path: &Path,
   ) -> Result<()> {
       tracing::info!(
           session = session_name,
           layout = %layout_path.display(),
           "Creating new Zellij session"
       );
       
       let status = Command::new("zellij")
           .args([
               "--session", session_name,
               "--layout", &layout_path.display().to_string(),
           ])
           .status()?;
       
       if !status.success() {
           anyhow::bail!(
               "Zellij session creation failed with exit code: {:?}",
               status.code()
           );
       }
       
       Ok(())
   }
   ```

2. Note: `zellij --session X --layout Y` starts a new session and attaches to it. When the user detaches, this process exits. This is the expected behavior — kasmos effectively "becomes" the Zellij session.

3. Alternative: use `std::os::unix::process::CommandExt::exec()` to replace the kasmos process entirely with Zellij. This is cleaner but platform-specific (Unix only — acceptable per constitution: Linux primary).

**Files**: `crates/kasmos/src/launch/session.rs`

**Purpose**: When already inside a Zellij session, create new tabs instead of a new session.

**Steps**:

   ```rust
   /// Create tabs in the current Zellij session.
   ///
   /// When running inside Zellij, we can't create a new session.
   /// Instead, we create tabs using the layout.
   pub fn create_tabs(layout_path: &Path) -> Result<()> {
       tracing::info!(
           layout = %layout_path.display(),
           "Creating tabs in current Zellij session"
       );
       
       // zellij action new-tab --layout <path>
       // This creates all tabs defined in the layout
       let status = Command::new("zellij")
           .args([
               "action", "new-tab",
               "--layout", &layout_path.display().to_string(),
           ])
           .status()?;
       
       if !status.success() {
           anyhow::bail!(
               "Zellij tab creation failed with exit code: {:?}",
               status.code()
           );
       }
       
       Ok(())
   }
   ```

2. **Important nuance** from architecture memory: Inside a Zellij session, use `zellij action <cmd>` (no `--session` flag). Outside, use `zellij --session X action ...`.

3. **Layout limitation**: `zellij action new-tab --layout` may only create one tab at a time. If the layout has 2 tabs, you may need to:
   - Generate two separate KDL files (one per tab)
   - Call `new-tab --layout` twice
   - Test this behavior with your Zellij version

### T040: Wire `launch/mod.rs` full flow

**Purpose**: Connect all launch components into the complete `kasmos [spec-prefix]` flow.

**Steps**:

1. Update `crates/kasmos/src/launch/mod.rs`:
   ```rust
   pub mod detect;
   pub mod layout;
   pub mod session;
   
   pub use detect::{detect_feature, list_features, FeatureInfo};
   
   use crate::config::KasmosConfig;
   use anyhow::{Context, Result};
   
   /// Main entry point for `kasmos [spec-prefix]`.
   pub async fn run(spec_prefix: Option<&str>, config: &KasmosConfig) -> Result<()> {
       // 1. Detect feature
       let specs_dir = std::env::current_dir()?.join(&config.paths.specs_dir);
       let feature = detect::detect_feature(spec_prefix, &specs_dir)?;
       
       let feature = match feature {
           Some(f) => f,
           None => {
               // TODO: Interactive feature selector
               // For now, list features and ask user to specify
               let features = detect::list_features(&specs_dir)?;
               if features.is_empty() {
                   anyhow::bail!("No feature specs found in {}", specs_dir.display());
               }
               println!("Available features:");
               for f in &features {
                   let status = match (f.has_spec, f.has_plan, f.has_tasks) {
                       (true, true, true) => "ready",
                       (true, true, false) => "planned",
                       (true, false, false) => "specified",
                       _ => "empty",
                   };
                   println!("  {} [{}]", f.slug, status);
               }
               anyhow::bail!("Please specify a feature: kasmos <spec-prefix>");
           }
       };
       
       tracing::info!(feature = %feature.slug, "Launching session");
       
       // 2. Generate manager command
       let manager_cmd = layout::build_manager_command(config, &feature.slug);
       
       // 3. Generate layout KDL
       let kdl = layout::generate_session_layout(config, &feature.slug, &manager_cmd)?;
       let layout_file = layout::write_layout_file(&kdl)?;
       
       // 4. Detect Zellij context and launch
       let session_name = format!("{}-{}", config.session.name, &feature.slug[..3]);
       
       match session::detect_zellij_context() {
           session::ZellijContext::Outside => {
               session::create_session(&session_name, layout_file.path())?;
           }
           session::ZellijContext::Inside { .. } => {
               session::create_tabs(layout_file.path())?;
           }
       }
       
       // 5. kasmos exits (fire-and-forget)
       tracing::info!("Session launched, kasmos exiting");
       Ok(())
   }
   ```

| Risk | Mitigation |
|------|-----------|
| KDL syntax errors in generated layout | Validate generated KDL by parsing with `kdl` crate before writing |

| Manager prompt escaping in KDL | Use shell-escape crate for prompt, proper KDL string escaping for args |
| Temp file cleanup | `NamedTempFile` auto-cleans on drop; ensure it lives long enough for Zellij to read |
