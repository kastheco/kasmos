# WP08: MCP Tools — Worker Management

## Implementation Command

```bash
spec-kitty implement WP08 --base WP07
```

## Objective

Implement three MCP tools — `spawn_worker`, `despawn_worker`, and `list_workers` — that give the manager agent full pane lifecycle control. After this WP, the manager can spawn OpenCode agents in named Zellij panes, track them in the in-memory registry, despawn them when done, and reconcile registry state with live pane reality.

## Context

- **Contracts**: `kitty-specs/011-mcp-agent-swarm-orchestration/contracts/kasmos-serve.json` — spawn_worker, despawn_worker, list_workers schemas

- `serve/mod.rs` has `KasmosServer` struct with `Arc<Mutex<WorkerRegistry>>` shared state
- `serve/tools/spawn_worker.rs`, `serve/tools/despawn_worker.rs`, `serve/tools/list_workers.rs` exist as stubs
- `serve/registry.rs` has `WorkerRegistry` with `register()`, `deregister()`, `get()`, `list_all()` methods
- `types.rs` has `AgentRole`, `WorkerEntry`, `WorkerStatus` types
- `config.rs` has `KasmosConfig` with `agent.binary`, `agent.profile`

### Key Design Decisions
- Manager is the ONLY agent with kasmos MCP access
- Workers communicate via msg-log pane (structured messages)
- Coder role gets a git worktree; planner/reviewer/release work in main repo
- Pane naming convention: `<wp_id>-<role>` (e.g., "WP-01-coder")
- Worker spawning is non-blocking (returns immediately after pane creation)

### T048: Implement `serve/tools/spawn_worker.rs`

**Purpose**: Replace the stub with a full implementation that creates a Zellij pane running an OpenCode agent and registers it in the worker registry.

**Steps**:
1. Define `SpawnWorkerInput` struct (derives `Deserialize`, `JsonSchema`):
   ```rust
   pub struct SpawnWorkerInput {
       pub wp_id: String,
       pub role: String,          // "planner", "coder", "reviewer", "release"
       pub prompt: String,
       pub cwd: Option<String>,
       pub context_files: Option<Vec<String>>,
   }
   ```

2. Implement the tool handler:
   - Parse `role` string → `AgentRole` enum (error if invalid)
   - Determine working directory:
     - If `cwd` is provided, use it
     - If role is `Coder`, create worktree (delegate to T050 function)
     - Otherwise, use repo root (from config or `std::env::current_dir()`)
   - Build the OpenCode command string (delegate to T049 function)
   - Create the Zellij pane via `zellij` CLI wrapper
   - Register the worker in the `WorkerRegistry`
   - Append audit entry (delegate to T053)
   - Return `SpawnWorkerOutput { pane_name, wp_id, role, cwd }`

3. Zellij pane creation command:
   ```bash
   zellij action new-pane --name "<wp_id>-<role>" --cwd <cwd> -- <opencode_command>
   ```
   - Note: If inside a Zellij session (which kasmos serve always is, since it runs in the MCP tab), use `zellij action` without `--session`
   - The `--` separator allows passing the full opencode command as the pane's process

4. Error handling:
   - Return error if `wp_id` is already registered (duplicate spawn)
   - Return error if Zellij command fails (capture stderr)
   - Log errors via `tracing::error!`

**Files**: `crates/kasmos/src/serve/tools/spawn_worker.rs` (~80 lines)

**Validation**:
- [ ] Spawning a worker creates a visible Zellij pane with the correct name
- [ ] Worker appears in the registry after spawn
- [ ] Duplicate spawn returns an error
- [ ] Pane runs the correct OpenCode command

### T049: Implement worker pane command construction

**Purpose**: Build the complete OpenCode CLI command string for a worker pane, handling shell escaping correctly.

**Steps**:
1. Create function in `spawn_worker.rs` or a shared util:
   ```rust
   fn build_opencode_command(
       config: &KasmosConfig,
       role: &AgentRole,
       prompt: &str,
       context_files: &[String],
   ) -> Vec<String>
   ```

2. Command structure:
   ```
   opencode -p <profile> -- --agent <role_flag> --prompt <prompt>
   ```
   Where:
   - `opencode` = `config.agent.binary` (default: "opencode")
   - `<profile>` = `config.agent.profile` (default: "kasmos")
   - `<role_flag>` = `role.agent_flag()` (from WP03: "manager"/"planner"/"coder"/"reviewer"/"release")
   - `<prompt>` = the initial prompt text (shell-escaped)

3. Shell escaping:
   - Use `shell_escape::escape()` for the prompt string (it may contain quotes, newlines, special chars)
   - Context files can be appended to the prompt as instructions: "Read the following files for context: <file1>, <file2>, ..."

4. Return the command as a `Vec<String>` of arguments (NOT a single string — Zellij's `new-pane --` takes individual args)

**Files**: `crates/kasmos/src/serve/tools/spawn_worker.rs` (add ~40 lines)

**Validation**:
- [ ] Generated command uses correct binary and profile from config
- [ ] Prompt with special characters (quotes, newlines) is properly escaped
- [ ] Role flag matches the agent role

### T050: Implement worktree creation for coder role

**Purpose**: When spawning a coder, create a git worktree so they work in isolation.

**Steps**:

   ```rust
   async fn ensure_worktree(
       config: &KasmosConfig,
       feature_slug: &str,
       wp_id: &str,
   ) -> Result<PathBuf, anyhow::Error>
   ```

2. Worktree path: `<config.paths.worktree_base>/<feature_slug>-<wp_id>/`
   - Example: `.worktrees/011-mcp-agent-swarm-orchestration-WP01/`

3. Logic:
   - Check if worktree already exists at the path → reuse if so

4. Reuse existing `git.rs` `WorktreeManager` if it has the needed functionality, or extend it:
   - `git.rs` already has `WorktreeManager` — check if it provides `add_worktree()`, `remove_worktree()`, `worktree_exists()`
   - If it does, delegate. If not, use `tokio::process::Command` to call `git worktree add` directly.

**Files**: `crates/kasmos/src/serve/tools/spawn_worker.rs` (add ~50 lines) or `crates/kasmos/src/git.rs` (extend)

### T051: Implement `serve/tools/despawn_worker.rs`

**Purpose**: Close a worker's Zellij pane and remove it from the registry.

**Steps**:
1. Define `DespawnWorkerInput`:
   ```rust
   pub struct DespawnWorkerInput {
       pub wp_id: String,
   }
   ```

2. Implementation:
   - Look up the worker in the registry by `wp_id`
   - If not found, return error: "Worker not found: {wp_id}"
   - Close the Zellij pane:
     - Use pane-tracker MCP or `zellij action` commands
     - The pane is named `<wp_id>-<role>` — use pane name for identification
     - Strategy: `zellij action close-pane` requires the pane to be focused first
     - Alternative: use `zellij-pane-tracker` if it has `close_pane_by_name`
   - Deregister from `WorkerRegistry`
   - Append audit entry
   - Return `DespawnWorkerOutput { success: true, wp_id }`

3. Edge cases:
   - Pane already closed (user manually closed it) → deregister anyway, return success
   - Capture and log zellij errors but still deregister

**Files**: `crates/kasmos/src/serve/tools/despawn_worker.rs` (~60 lines)

**Validation**:
- [ ] Despawning a registered worker removes it from registry
- [ ] Despawning a non-existent worker returns an error
- [ ] If the pane is already gone, despawn still succeeds (cleanup)
- [ ] Audit entry is written

### T052: Implement `serve/tools/list_workers.rs`

**Purpose**: Return all workers with their current status, reconciled against live Zellij pane state.

**Steps**:
1. Implementation:
   - Get all workers from the `WorkerRegistry` via `list_all()`
   - Get live pane list from Zellij (via `zellij action list-panes` or equivalent — note: standard Zellij CLI may not have this. Check `zellij.rs` for available wrappers)
   - Reconciliation logic:
     - For each registered worker, check if its pane is still alive
     - If pane exists → status stays as-is (Active, Done, etc.)
     - If pane is gone AND status is Active → mark as `Aborted`
     - If pane is gone AND status is Done → leave as Done (expected)
   - Calculate `elapsed_seconds` from `worker.spawned_at` to now

2. Return format:
   ```json
   {
     "workers": [
       {
         "wp_id": "WP-01",
         "role": "coder",
         "pane_name": "WP-01-coder",
         "status": "active",
         "elapsed_seconds": 142
       }
     ]
   }
   ```

3. Pane detection approach:
   - `zellij action dump-screen` or `zellij action query-tab-names` may not give pane names
   - **Pragmatic approach**: Use `zellij-pane-tracker` MCP server's `get_panes` tool if available (it's registered in the manager's OpenCode profile)
   - **Alternative**: Shell out to `zellij action list-panes` (if the Zellij version supports it) and grep for pane names
   - **Fallback**: Track pane liveness by attempting to dump the pane and checking for errors

**Files**: `crates/kasmos/src/serve/tools/list_workers.rs` (~70 lines)

**Validation**:
- [ ] Returns all registered workers with correct fields
- [ ] Detects and marks aborted workers (pane gone while active)
- [ ] Elapsed time is calculated correctly
- [ ] Empty registry returns empty array

### T053: Add audit logging to spawn/despawn operations

**Purpose**: Record all worker lifecycle events in the JSONL audit log.

**Steps**:
1. In `spawn_worker.rs`, after successful registration:
   ```rust
   AuditEntry::append(
       &config.paths.audit_dir,
       AuditEntry {
           timestamp: Utc::now(),
           actor: "kasmos-serve".to_string(),
           action: "spawn_worker".to_string(),

               "wp_id": input.wp_id,
               "role": input.role,
               "pane_name": pane_name,
               "cwd": cwd_str,
           }),
       },
   ).await?;
   ```

2. In `despawn_worker.rs`, after successful deregistration:
   ```rust
   AuditEntry::append(
       &config.paths.audit_dir,

           timestamp: Utc::now(),

           action: "despawn_worker".to_string(),
   ```

4. Ensure the `.kasmos/` directory is created if it doesn't exist (create_dir_all)

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| `zellij action new-pane` argument format varies by version | Pin Zellij 0.43.x+, test exact CLI syntax |
| Pane name collision if same WP respawned | Check registry first, return error on duplicate |
| Shell escaping of prompts | Use `shell_escape` crate consistently |
| Worktree creation race condition | Check existence before creating, handle "already exists" gracefully |
