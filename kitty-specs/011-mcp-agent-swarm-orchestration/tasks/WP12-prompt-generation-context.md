# WP12 — Prompt Generation & Context Assembly

## Goal

Adapt `crates/kasmos/src/prompt.rs` to generate role-specific OpenCode launch commands with context file assembly per agent role. The rewritten module produces `AgentCommand` structs consumed by `spawn_worker` to construct the correct `opencode -p kasmos -- --agent <role> --prompt <text>` invocation with appropriate context.

## Dependencies

| WP | What it provides |
|----|-----------------|
| WP08 | `spawn_worker` tool in `serve/tools/worker_management.rs` — consumes prompt output |
| WP11 | Agent profiles in `config/profiles/kasmos/` — defines role names, MCP servers, instruction files |
| WP03 | `AgentRole` enum in `serve/types.rs` |
| WP02 | `KasmosConfig` with `[agent]` section (binary, profile) |

## Subtasks

### T073 — Rewrite `prompt.rs` core interface

**Current state** (`crates/kasmos/src/prompt.rs`, 435 lines):
- `PromptContext` struct with `render()` → markdown string
- `PromptGenerator` — loads AGENTS.md, writes prompt files to disk, generates shell scripts
- `validate_binary_in_path()` — uses `which` crate
- Shell scripts hardcode `ocx oc -- --agent coder --prompt "$(cat '...')"` (line 252)
- Placeholder subtask extraction (single dummy subtask)
- Hardcoded feature name "001-zellij-agent-orchestrator"

**What to do**:

1. Replace `PromptGenerator` and `PromptContext` with a new function-based API:

```rust
use crate::serve::types::AgentRole;
use crate::config::KasmosConfig;

/// Fully-resolved command to launch an agent in a pane.
pub struct AgentCommand {
    /// Binary to execute (e.g., "opencode")
    pub binary: String,
    /// Arguments to pass (e.g., ["-p", "kasmos", "--", "--agent", "coder", "--prompt", "..."])
    pub args: Vec<String>,
    /// Working directory for the pane
    pub cwd: PathBuf,
    /// Pane name for Zellij
    pub pane_name: String,
}

/// Generate a complete agent launch command for a given role and context.
pub fn generate_command(
    role: AgentRole,
    wp_id: &str,
    feature_slug: &str,
    config: &KasmosConfig,
    cwd: &Path,
) -> Result<AgentCommand>;
```

2. **Keep** `validate_binary_in_path()` — it's still needed by setup and launch.
3. **Remove** `PromptGenerator`, `PromptContext`, `SubtaskInfo`, `DependencyContext` structs.
4. **Remove** `generate_all()`, `generate_scripts()`, `build_prompt_context()`, `build_dependency_context()`, `infer_key_outputs()`.
5. **Remove** all shell script generation — no longer needed (MCP `spawn_worker` handles invocation).
6. Update `lib.rs` exports for the new API.
7. Update or remove tests that reference deleted structs. Write new tests for `generate_command()`.

**Binary construction**: Use `config.agent.binary` (default "opencode"), `config.agent.profile` (default "kasmos"):
```
opencode -p kasmos -- --agent {role} --prompt "{prompt_text}"
```

### T074 — Context file assembly per role

Implement a function that determines which context files each role should be instructed to read. The prompt text will include explicit `Read these files:` instructions rather than inlining content.

```rust
/// Files the agent should read as initial context.
pub fn context_files_for_role(
    role: AgentRole,
    wp_id: &str,
    feature_slug: &str,
    project_root: &Path,
) -> Vec<PathBuf>;
```

**Context file matrix** (from plan.md Agent Profiles section):

| Role | Context files |
|------|--------------|
| Manager | `kitty-specs/{feature}/spec.md`, `kitty-specs/{feature}/plan.md`, `kitty-specs/{feature}/tasks.md`, `.kittify/memory/constitution.md`, `.kittify/memory/architecture.md`, `AGENTS.md` |
| Planner | `kitty-specs/{feature}/spec.md`, `kitty-specs/{feature}/plan.md` (if exists), `.kittify/memory/constitution.md`, `.kittify/memory/architecture.md`, `AGENTS.md` |
| Coder | `kitty-specs/{feature}/tasks/{wp_id}-*.md` (the WP prompt file), `.kittify/memory/constitution.md`, `AGENTS.md` |
| Reviewer | `kitty-specs/{feature}/tasks/{wp_id}-*.md`, `AGENTS.md` |
| Release | `kitty-specs/{feature}/tasks.md`, `kitty-specs/{feature}/plan.md`, `AGENTS.md` |

**Rules**:
- Only include files that exist (skip missing optional files like architecture.md).
- Use paths relative to project root for portability.
- Coder's context is intentionally minimal — scoped to their WP only.

**Tests**: Verify correct file lists per role, verify missing files are skipped gracefully.

### T075 — Initial prompt text generation per role

Implement prompt text templates that instruct each role on their task and communication protocol.

```rust
/// Generate the initial prompt text for an agent.
fn prompt_text_for_role(
    role: AgentRole,
    wp_id: &str,
    feature_slug: &str,
    context_files: &[PathBuf],
    config: &KasmosConfig,
) -> String;
```

**Template structure per role**:

**Coder prompt**:
```
You are implementing work package {wp_id} for feature {feature_slug}.

Read these context files first:
{context_files, one per line}

Your working directory is a git worktree. All changes stay in this worktree.

When you reach milestones, write structured messages to the msg-log pane using the zellij MCP:
- Start: run_in_pane(pane="msg-log", command="echo '[KASMOS:{wp_id}:STARTED] Beginning implementation'")
- Progress: run_in_pane(pane="msg-log", command="echo '[KASMOS:{wp_id}:PROGRESS] {description}'")
- Done: run_in_pane(pane="msg-log", command="echo '[KASMOS:{wp_id}:DONE] Implementation complete'")
- Error: run_in_pane(pane="msg-log", command="echo '[KASMOS:{wp_id}:ERROR] {description}'")

Begin by reading the WP task file, then implement each subtask in order.
```

**Reviewer prompt**:
```
You are reviewing work package {wp_id} for feature {feature_slug}.

Read these context files first:
{context_files, one per line}

Review the changes in the worktree against the WP acceptance criteria.

Report your verdict to the msg-log pane:
- Pass: run_in_pane(pane="msg-log", command="echo '[KASMOS:{wp_id}:REVIEW_PASS] {summary}'")
- Reject: run_in_pane(pane="msg-log", command="echo '[KASMOS:{wp_id}:REVIEW_REJECT] {feedback}'")
```

**Manager prompt**: (more complex — includes wait_for_event loop pattern, delegation instructions, confirmation gates — reference the plan.md Manager profile section)

**Planner prompt**: (includes spec-kitty lifecycle instructions, msg-log completion signal)

**Release prompt**: (includes merge procedure, acceptance test instructions, cleanup)

**Rules**:
- Keep prompts under 2000 characters where possible (OpenCode stdin has practical limits).
- Reference files rather than inlining content.
- Include the structured message format `[KASMOS:{wp_id}:{EVENT}]` literally — workers must use this exact prefix.
- Use `config.communication.message_prefix` (default "KASMOS") for the prefix.

**Tests**: Verify prompt text contains required sections, verify message format is present, verify context files are listed.

### T076 — Wire prompt generation into `spawn_worker`

Connect `generate_command()` to the `spawn_worker` MCP tool implemented in WP08.

**Current spawn_worker** (from WP08): Takes `wp_id`, `role`, and `prompt` parameters. It creates the pane and runs the agent.

**Integration point**: In `serve/tools/worker_management.rs`, the `spawn_worker` handler should:

1. Call `generate_command(role, wp_id, feature, config, cwd)` to get the `AgentCommand`.
2. Use `AgentCommand.binary`, `AgentCommand.args`, `AgentCommand.cwd`, `AgentCommand.pane_name` to create the Zellij pane and run the agent.
3. The `prompt` parameter in the MCP tool becomes optional — if provided, it overrides the generated prompt (escape hatch for manager).

**What to change in spawn_worker**:
- Import `crate::prompt::generate_command`
- Replace hardcoded command construction with `generate_command()` call
- Keep the override path: if `prompt` is provided in MCP call, use it directly

**Tests**: Integration test that `spawn_worker` with `role=coder, wp_id=WP01` produces the correct command string.
